# Отчет по модульному тестированию

## 1. Выполненная работа

В проект добавлен автоматизированный набор модульных тестов для модулей:
- `program_files/auth.py`
- `program_files/database.py`
- `program_files/api_client.py`
- `program_files/timer.py`

Реализовано **124 модульных теста** (`tests/`), что покрывает требование не менее 100 тестов для команды из 4 человек (минимум 25 на человека).

## 2. Использованные инструменты

- `pytest` как фреймворк тестирования
- `pytest-cov` для оценки покрытия
- `unittest.mock`/`pytest.monkeypatch` для mock/stub
- `sqlite3` с изолированной тестовой БД через фикстуры
- GitHub Actions для автоматического запуска тестов

## 3. Примененные техники тест-дизайна

- Классы эквивалентности:
  - валидные/невалидные логины и пароли
  - успешные/неуспешные ответы внешних API
- Граничные условия:
  - таймер на границе завершения сессии (`remaining <= 0`)
  - длительность сессии (`1`, `30`, `1500`, `3600`)
  - фильтрация сессий по месяцу/году
- Попарное тестирование:
  - комбинации параметров уроков в расписании (`auditories`, `teachers`, `typeObj`, `lms_url`)
  - комбинации состояний таймера (`running`/`app_running`)

## 4. Автоматизация при сборке

Тесты интегрированы в CI-пайплайн: [`.github/workflows/ci.yml`](../.github/workflows/ci.yml).

Запуск происходит:
- по событию изменения кода (`push`, `pull_request`)
- по расписанию (`cron`)

В job `unit-tests` выполняется:
```bash
pytest tests --cov=program_files --cov-fail-under=80 --cov-report=xml --junitxml=unit-junit.xml
```

## 5. Отчет о прохождении и покрытии

- JUnit-отчет: `unit-junit.xml`
- Coverage XML: `coverage.xml`
- Порог покрытия: `--cov-fail-under=80`

Фактическое значение покрытия определяется запуском пайплайна. Если покрытие ниже 80%, сборка завершается с ошибкой.

## 6. Процедура расширения тестового набора

Пример: добавляется новый метод `Database.delete_session(session_id)`.

1. Добавить позитивный тест:
   - создается пользователь
   - создается сессия
   - вызывается `delete_session`
   - проверяется отсутствие сессии в `get_user_sessions`
2. Добавить негативный тест:
   - удаление несуществующей `session_id`
   - метод должен корректно отработать без падения
3. Добавить граничный тест:
   - удаление последней сессии пользователя
4. Запустить:
```bash
pytest tests --cov=program_files --cov-fail-under=80
```
5. При необходимости обновить интеграционный сценарий, если метод участвует в пользовательском процессе.
