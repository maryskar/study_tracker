# Отчет по интеграционному тестированию

## 1. Документ с тестовыми сценариями

Набор интеграционных сценариев реализован в файле:
- `integration_tests/test_integration_workflows.py`

Всего сценариев: **10** (минимум по требованию выполнен).

## 2. Перечень сценариев

1. Регистрация -> логин -> старт/стоп `pomodoro` -> проверка записи сессии в БД.
2. Негативный сценарий: повторная регистрация того же пользователя.
3. Негативный сценарий: логин с неверным паролем.
4. Запуск `stopwatch` и сохранение сессии.
5. Завершение `pomodoro` и проверка выдачи достижения.
6. Интеграция с mock внешних API (мотивация, время, расписание) + запуск сессии.
7. Негативный сценарий: fallback при таймауте внешних API.
8. Проверка изоляции данных по пользователям при выборке по месяцу.
9. Последовательный запуск двух разных сессий одним пользователем.
10. Негативный сценарий: вызов `stop_timer` без активной сессии.

## 3. Заглушки и изоляция

Использованы mock-объекты и заглушки:
- `patch("program_files.api_client.requests.get")` для внешних HTTP API
- `InlineScheduler` для изоляции планировщика таймера
- отдельная временная SQLite БД на каждый сценарий через фикстуры

Это ускоряет тесты и устраняет зависимость от сети/внешнего окружения.

## 4. Негативные сценарии

Покрыты сценарии отказа:
- неуспешная аутентификация
- дублирование пользователя
- недоступность внешних сервисов
- вызов остановки таймера без предварительного запуска

## 5. CI запуск интеграционных тестов

Интеграционные тесты запускаются в GitHub Actions job `integration-tests`:

```bash
pytest integration_tests -m integration --junitxml=integration-junit.xml
```

Триггеры запуска:
- изменения кода (`push`/`pull_request`)
- временной триггер (`schedule` в workflow)

## 6. Артефакты

CI формирует и сохраняет:
- `integration-junit.xml` (результаты сценариев)
- `unit-junit.xml`, `coverage.xml` (модульное тестирование)

Файл workflow: `.github/workflows/ci.yml`
